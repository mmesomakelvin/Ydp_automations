# 4. YDP Program Control Center

## Plain-English Goal

Build an internal website where the YDP team can see what automations, documents, dashboards, and build tasks exist in one place.

This becomes the team's single source of truth for:

- what has been built,
- what is still being designed,
- what is blocked,
- where the documentation lives,
- where the Google Sheets, Apps Script projects, dashboards, and GitHub files are.

## Current Status

Status: **Brainstorming/design in progress**

No implementation has started yet.

## Confirmed Decisions

### Portal Name

The internal portal will be called:

```text
YDP Program Control Center
```

### Hosting

- The portal will be hosted on Vercel.
- It is for the internal team only.

### Access Control

Recommended and accepted direction:

- Use Google login.
- Check the signed-in user's email against an allowlist.
- Store the allowlist in a Vercel environment variable for version 1.

Example environment variable:

```text
ALLOWED_EMAILS=person1@example.com,person2@example.com,person3@example.com
```

Why this approach:

- More secure than one shared password.
- Easier than building a full admin access system immediately.
- Team access can be changed without editing code.
- Later, the allowlist can move into Google Sheets if needed.

### Homepage Direction

The homepage should combine:

1. **Automation status dashboard**
   - Shows what is built, in progress, pending, or blocked.

2. **Documentation library**
   - Shows links to artifacts, specs, Apps Script projects, sheets, dashboards, and GitHub files.

This gives the team a quick operational view and a documentation hub in the same place.

### Data Source

Use a hybrid source-of-truth model:

| Source | Purpose |
| --- | --- |
| Google Sheet | Live tracker for statuses, owners, priorities, links, next actions, due dates |
| GitHub Markdown artifacts | Detailed documentation, design notes, specs, implementation notes |
| Vercel website | Internal portal that displays both in a clean interface |

### Tracker Structure

The Google Sheet tracker should have both automation-level and task-level tracking.

Recommended tabs:

| Tab | Purpose |
| --- | --- |
| Automations | One row per automation/system/superpower |
| Tasks | One row per build task or action item |
| Decisions | Important decisions made during planning |
| Links | Apps Script, Sheets, GitHub, dashboard, and documentation links |
| Access | Optional later if access control moves from Vercel env var to Google Sheets |

### Status Labels

Use detailed statuses:

| Status | Meaning |
| --- | --- |
| Idea | Mentioned but not scoped |
| Brainstorming | We are asking questions and shaping the system |
| Designed | Spec/design is approved |
| Building | Implementation is in progress |
| Testing | Built but being tested |
| Deployed | Live and usable |
| Blocked | Waiting on access, decision, data, API, or another dependency |

### Editing Model

Version 1 should be:

```text
View-only portal first, editable later.
```

Meaning:

- Team members edit the Google Sheet tracker.
- The Vercel site reads and displays the tracker.
- The website does not update tracker data in version 1.

This keeps the first version simpler and safer.

## Recommended Homepage Sections

| Section | What it shows |
| --- | --- |
| KPI Cards | Total automations, deployed, building, brainstorming, blocked |
| Automation Table | Name, status, priority, owner, last updated, next action |
| Documentation Library | Links to artifacts/specs grouped by automation |
| Quick Links | Google Sheets, Apps Script projects, dashboards, GitHub repo |
| Recent Updates | Latest completed work or changed statuses |
| Open Decisions | Questions that still need answers |
