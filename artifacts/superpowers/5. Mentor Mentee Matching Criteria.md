## Plain-English Goal

Create a fair, explainable way to select mentees and match selected mentees to mentors.

The matching system should not just pair people randomly. It should use the application data to understand:

- whether the mentee is ready for mentorship,
- what the mentee wants to learn,
- what the mentor can support,
- how many mentees each mentor can take,
- whether the match makes practical sense.

## Mentee Selection Criteria

Before matching, mentees should be scored so the team knows who is most ready for the program.

| Criteria | Weight | What it means |
| --- | --- | --- |
| Learning Commitment & Prior Effort | 40% | Has the mentee actively learned, taken courses, built projects, or practiced recently? |
| Community Engagement & Participation | 20% | Is the mentee active in the YDP community or related activities? |
| Career Goals & Program Alignment | 20% | Does the mentee have clear goals that mentorship can realistically support? |
| Soft Skills & Mindset | 20% | Is the mentee proactive, open to feedback, and willing to do the work? |

Each category is scored from `0` to `5`, then weighted into a final score out of `100`.

## Mentee Selection Scoring

### Learning Commitment & Prior Effort

Weight: **40%**

Score guide:

| Score | Meaning |
| --- | --- |
| 5 | Strong evidence of learning: multiple courses, projects, certifications, or real-world applications |
| 4 | Moderate engagement: 1-2 courses, small projects, or learning with some application |
| 3 | Some effort: learning but minimal hands-on application |
| 2 | Limited effort: some exposure but no serious learning |
| 1 | Very minimal effort, no real progress |
| 0 | No learning effort, expects to be spoon-fed |

### Community Engagement & Participation

Weight: **20%**

Score guide:

| Score | Meaning |
| --- | --- |
| 5 | Highly engaged: active in discussions, volunteering, events, collaborations |
| 4 | Moderately engaged: participates occasionally, attends events, some interaction |
| 3 | Somewhat engaged: member but not very active |
| 2 | Minimal engagement: joined but never participated |
| 1 | No engagement but aware of the community |
| 0 | Not a member of YDP |

### Career Goals & Program Alignment

Weight: **20%**

Score guide:

| Score | Meaning |
| --- | --- |
| 5 | Very clear career goals, well-defined learning expectations, strong mentorship alignment |
| 4 | Somewhat clear goals, has an idea of skills needed but lacks specifics |
| 3 | General idea but unclear plan for applying mentorship learnings |
| 2 | Vague goals, unsure about career direction |
| 1 | No clear career goal |
| 0 | No response or unrealistic expectations from mentorship |

### Soft Skills & Mindset

Weight: **20%**

Score guide:

| Score | Meaning |
| --- | --- |
| 5 | Highly proactive, problem-solver, takes initiative, respects mentorship value |
| 4 | Some initiative, willing to learn, but needs guidance |
| 3 | Open to learning but hesitant to take action |
| 2 | Passive learner, expects step-by-step guidance |
| 1 | Expects to be spoon-fed, lacks motivation |
| 0 | No growth mindset, dependency mentality |

## Final Mentee Score Formula

Because the category scores are from `0` to `5`, the weighted formula is:

```text
Final Score =
  Learning Commitment Score * 8
  + Community Engagement Score * 4
  + Career Goals Score * 4
  + Soft Skills Score * 4
```

Maximum score:

```text
100
```

Example:

| Category | Score | Weighted Score |
| --- | --- | --- |
| Learning Commitment | 4/5 | 32 |
| Community Engagement | 3/5 | 12 |
| Career Goals & Fit | 5/5 | 20 |
| Soft Skills & Mindset | 4/5 | 16 |

Final score:

```text
80/100
```

## Selection Rules

- Final score determines mentee selection priority.
- A cutoff score should be set based on available mentor capacity.
- If applicants tie, priority should go to:
  - earlier applicants,
  - active YDP members.

## Mentor-Mentee Pairing Criteria

Once mentees are selected, pairing should use the following criteria.

| Priority | Criteria | Importance |
| --- | --- | --- |
| 1 | Skills & Career Path Alignment | Must-have |
| 2 | Mentor Availability | Must-have |
| 3 | Experience Level Consideration | Highly recommended |
| 4 | Industry Relevance | Optional but useful |
| 5 | Learning Preferences | Nice-to-have |
| 6 | Time Zone Compatibility | Nice-to-have |

## Pairing Rules In Plain English

### 1. Skills & Expertise Matching

Match mentees with mentors who can support the skills the mentee wants to learn.

Example:

```text
Mentee wants SQL + Power BI
Best mentor has SQL + Power BI + Data Visualization
```

### 2. Career Path Alignment

Match mentees with mentors who have experience in the career path the mentee is pursuing.

Example:

```text
Mentee wants Data Engineering
Best mentor is a Data Engineer or has strong Data Engineering experience
```

### 3. Industry Relevance

If the mentee prefers an industry, try to match them with a mentor from that industry.

Example industries:

- finance,
- healthcare,
- e-commerce,
- technology.

### 4. Experience Level Consideration

Senior mentors can support a wider range of mentees.

Mid-level mentors should usually be paired with early-career mentees.

### 5. Mentor Availability

Mentors have limits on how many mentees they can take.

The matching system must respect mentor capacity.

Example:

```text
If a mentor can take 3 mentees, do not assign 5 mentees to them.
```

### 6. Communication & Learning Preferences

Where possible, match learning style with mentor style.

Example:

```text
Hands-on mentee -> mentor who supports practical/project-based learning
Structured learner -> mentor who gives detailed explanations and plans
```

### 7. Time Zone & Meeting Availability

If available, use overlapping availability to avoid impractical matches.

This is helpful but should not override skill/career alignment.

## Recommended Automation Approach

The current decision is to auto-match first, then let the team review afterward.

Version 1 should:

1. Score mentees using the selection criteria.
2. Score every eligible mentee against every available mentor in `Pair Scores`.
3. Use Gemini for pair scoring across skill fit, career fit, availability fit, and capacity fit.
4. Auto-select the best mentor for each mentee from saved pair scores.
5. Respect mentor flexible capacity.
6. Write selected matches into `Match Recommendations`.
7. Write final assignments into `Matched Pairs`.
8. Mark matches as `Auto-Matched` and `Pending Review`.

This avoids waiting on manual approval while keeping the decision trail reviewable.

The workbook setup creates:

- `Source Config`
- `Mentee Source Snapshot`
- `Mentor Source Snapshot`
- `Mentee Scores`
- `Pair Scores`
- `Match Recommendations`
- `Matched Pairs`
- `Run Log`

The current build sets up the structure, imports source snapshots, and tests the Gemini connection.

In plain English:

- `Setup matching workbook` creates the tabs and configuration area.
- `Sync source snapshots from forms` copies the latest rows from the live mentee and mentor form response sheets into this matching workbook.
- `Generate next mentee score` scores one unscored mentee using the YDP selection criteria and writes the result into `Mentee Scores`.
- `Generate mentee scores batch` scores up to 5 unscored mentees using the same scoring logic.
- `Test Gemini connection` checks that Gemini can be called from the matching script.

The snapshot tabs are expected to be blank after setup only. They should fill after `Sync source snapshots from forms` runs successfully.

The sync step can auto-detect source response tabs named like `Form Responses 1`, even if the config still says `Form_Responses`.

The scoring step uses Gemini to read free-text answers and return structured category scores. It also writes `Gemini Review Status` so the automation knows who crossed the threshold for pairing.

Use `Generate next mentee score` first as a one-person test. After that works, use `Generate mentee scores batch` to move faster. Completed scores are preserved. Already-scored mentees are skipped. Rows marked `ERROR` are retried. If quota is reached, the script stops safely and the team can run the same button again later to continue.

The next build will add pair scoring and auto-matching.

## Pair Scores

`Pair Scores` stores every Gemini comparison between one eligible mentee and one available mentor.

Eligible mentees are the ones marked `Gemini Review Status = Can Pair` in `Mentee Scores`. Mentees marked `Do Not Pair` are not sent into pair scoring.

Example:

| Mentee | Mentor | Total Pair Score |
| --- | --- | ---: |
| Deborah | Israel | 87 |
| Deborah | Jerry | 77 |
| Deborah | Rume | 65 |

Deborah is still assigned to only one mentor. The extra rows are comparison evidence.

Pair score weighting:

| Category | Weight |
| --- | ---: |
| Skill fit | 40 |
| Career fit | 30 |
| Availability fit | 15 |
| Capacity fit | 15 |
| Total | 100 |

## Auto-Matching Output

The system uses `Pair Scores` to produce `Match Recommendations`:

| Header | Meaning |
| --- | --- |
| Mentee ID | The selected mentee |
| Mentee Name | Readable name |
| Mentee Career Path | Target role/path |
| Recommended Mentor ID | Best mentor candidate |
| Recommended Mentor Name | Readable name |
| Match Score | Overall score for the pair |
| Skill Match Reason | Why the skill match makes sense |
| Career Match Reason | Why the career path fit makes sense |
| Availability Notes | Any availability/capacity information |
| Review Status | Pending, Approved, Rejected |
| Final Decision Notes | Human reviewer notes |

Then it writes final assignments into `Matched Pairs`.

Each final match starts as:

```text
Match Status = Auto-Matched
Review Status = Pending Review
```

## Mentor Capacity Rule

Flexible mentor capacity is:

```text
Stated mentor capacity + 2
```

If a mentor says they can take `3` mentees, the system may assign up to `5`.

The system should prefer staying within stated capacity first, but may use the extra `+2` where needed.
