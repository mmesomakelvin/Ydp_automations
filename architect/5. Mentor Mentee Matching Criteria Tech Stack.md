# 5. Mentor-Mentee Matching Criteria Tech Stack

## Purpose

Create an explainable matching workflow that scores mentee readiness and recommends mentor-mentee pairings using application data.

## Version 1 Stack

| Layer | Tool | Why |
| --- | --- | --- |
| Source documentation | Google Doc | Contains the original selection and pairing criteria. |
| Source data | Mentee and mentor Google Sheets | Application responses already live there. |
| Stable identity | `Mentee ID` and `Mentor ID` columns | Prevents confusion when names repeat or someone submits more than once. |
| Scoring workspace | Google Sheets | Lets the team inspect mentee scores, pair scores, recommendations, and final matches. |
| Automation runtime | Google Apps Script V8 | Reads form data, creates scoring columns, scores mentor/mentee pairs, and generates final matches. |
| AI pair scoring | Gemini API | Scores mentor/mentee pair fit and explains match reasoning. |
| Human review | Google Sheets review columns | The team reviews after the system creates matches and can override manually. |
| Final output | `Matched Pairs` sheet | Official source of active program pairings. |
| Version control | GitHub | Tracks criteria, scripts, and architecture docs. |

## Initial Build Scope

The first implementation creates:

- `Source Config`
- `Mentee Source Snapshot`
- `Mentor Source Snapshot`
- `Mentee Scores`
- `Pair Scores`
- `Match Recommendations`
- `Matched Pairs`
- `Run Log`
- a menu named `YDP Matching`
- a Gemini connection test
- source snapshot syncing from the mentee and mentor response sheets

## Recommended Workflow

1. Assign unique IDs to mentors and mentees.
2. Score mentees using the selection criteria.
3. Select eligible mentees with `Final Score >= 60` and `Review Status = Pending Review`.
4. Generate `Pair Scores` for each eligible mentee against each available mentor.
5. Auto-select the best available mentor for each mentee from saved pair scores.
6. Write selected decisions into `Match Recommendations`.
7. Write final assignments into `Matched Pairs`.
8. Let the team review and override after auto-matching.

## Scoring Rules

Mentee selection should use:

| Criteria | Weight |
| --- | --- |
| Learning Commitment & Prior Effort | 40% |
| Community Engagement & Participation | 20% |
| Career Goals & Program Alignment | 20% |
| Soft Skills & Mindset | 20% |

Each category is scored from `0` to `5`.

Final score:

```text
Learning * 8 + Community * 4 + Career * 4 + Mindset * 4
```

## Matching Rules

Priority order:

1. Skills and career path alignment.
2. Mentor availability/capacity.
3. Experience level fit.
4. Industry relevance.
5. Learning preferences.
6. Time zone and meeting availability.

## Pair Scoring Rules

Gemini scores each mentor/mentee pair using:

| Category | Weight |
| --- | ---: |
| Skill fit | 40 |
| Career fit | 30 |
| Availability fit | 15 |
| Capacity fit | 15 |
| Total | 100 |

Pair scoring must be resumable because Gemini quota is limited:

- score one pair per run,
- skip already-scored pair IDs,
- use Script Property key slots `GEMINI_API_KEY`, `GEMINI_API_KEY_2`, and `GEMINI_API_KEY_3`,
- move to the next configured key only when Gemini reports a quota or rate limit,
- remember the last working slot in `GEMINI_ACTIVE_API_KEY_INDEX`,
- stop safely when quota is reached,
- preserve existing pair scores.

Key slot ownership is documented in the root README. Secret key values must remain in Apps Script Script Properties and must never be committed to GitHub.

## Capacity Rule

Flexible mentor capacity:

```text
mentor stated capacity + 2
```

## Do Not Add In Version 1

- Matching based only on names or manual memory.
- Matching applicants who were not selected into the program.
- Over-assigning mentors beyond flexible capacity.
- Sending final match emails before the team reviews the first output.
